# 章节交付前检查清单

> 来源：`docs/笔法核心规则.md`
> 使用方法：每章写完后，先跑 `python scripts/lint_chapter.py chapters/新_第XX章_标题.md`，再对照本清单逐项自检。

---

## 一、可由脚本自动检查（必须全绿）

运行：

```bash
python scripts/lint_chapter.py chapters/新_第XX章_标题.md
```

检查项：

- [ ] 每段不超过 3 句话（情绪戏可放宽到 4 句，但须标注原因）
- [ ] 每章「不是……是……」结构不超过 3 处
- [ ] 每章禁用过渡词（然后/一会儿/忽然/接着/之后）不超过 2 处（对话中的口语化"然后"可保留）
- [ ] 每章破折号（—）不超过 3 处
- [ ] 每章「像是在/仿佛在」不超过 2 处

---

## 二、必须人工自检（无法自动化）

### 篇章结构

- [ ] **对话驱动**：主要情节靠对话推进，信息、冲突、人物关系通过对话带出，而非旁白叙述。
- [ ] **无大段背景描写**：背景信息揉碎在剧情里，一章最多给三五句背景。
- [ ] **章末有钩子**：章尾必须让读者想翻页（出现新人、没说完的话、反常细节、悬而未决的决定）。

### 语言风格

- [ ] **不写心理描写，写动作**：没有"他心里很难过/很紧张/很害怕"，情绪通过身体动作传达。
- [ ] **不直接解释动作和现象**：删掉"那是他紧张时的微表情""他懂了""意思是/这说明""叙事者总结"等作者跳出来翻译的句子。
- [ ] **主角内心现代口语每章限 2-3 处**：只在情绪最外露时使用。
- [ ] **写感情用具体选择和动作**，不让角色表白或内心承认。
- [ ] **写悲情只写事实**，不替读者哭。

### 人物

- [ ] **许若存开朗、嘴贫、有脾气**：遇到事情第一反应是说话，不是沉默。
- [ ] **每个出场人物有一个记忆点**：一个习惯动作、一个矛盾点、一种独特的说话方式。
- [ ] **人物台词遮住名字能认出是谁**。
- [ ] **世界观通过角色反映，不通过角色解释**：角色台词不是 FAQ 界面。
- [ ] **人物不可预判**：角色立场、阵营、动机通过选择和行动逐步揭晓，不是靠出场标签。

### 节奏与结构

- [ ] **每章至少一处必须现在解决的冲突**：有时间压力、可见代价、不可逆选择。
- [ ] **场景之间不写过渡**：上一场最后一句直接空行，接下一个场景第一句，不用"与此同时""另一边"。
- [ ] **第一句入戏**：开头直接是动作/声音/触感/对话/反常画面，不写"从XX出发""沿着XX走了一炷香"。
- [ ] **节奏变速**：动作/逃跑/战斗用短句，安静/情绪场景句子稍长，悬疑紧张越来越短。
- [ ] **砍掉内心总结**：章尾/中间没有"把今天的事过了一遍"，没有分析他人动机，没有主题升华。
- [ ] **对话有信息差**：每段对话里至少一方在冒风险或试探，不是双方都说客观事实。

### 玄幻与战斗（第10章起）

- [ ] **每章至少三处能力/法宝/修为在实战中激活**。
- [ ] **追兵有修行质感**：武器、身体、招式、感知、配置至少有一处体现。
- [ ] **法宝使用有消耗和代价**：每次使用写清楚消耗了什么。
- [ ] **前后章追兵不重复**：相邻两章威胁类型不同。
- [ ] **战斗打得爆、节奏密、打完有变化**。

### 主题与影射

- [ ] **不讲大道理，观点融入故事**。
- [ ] **不借角色之口喊口号**，让读者自己品。
- [ ] **每个正面角色有缺陷，每个反面角色有理由**。

### 叙事视角

- [ ] **主角的信息边界**：叙述者不代主角认人；知识必须有来源；内心独白和对话不提前引用本章还没出现的信息。
- [ ] **他人的知情权**：其他角色知道主角有玉佩/铜钱时，必须有明确信息来源。
- [ ] **主角经常理解错**：不让他连续两次猜对，适当增加一次误判。
- [ ] **信息挤牙膏式释放**：同一信息分 3-5 块释放，一章最多揭 2/3，留 1/3 延后。

### 写后自查（第21节）

- [ ] **字数**：情节推进章 4500-5500，大战/多线冲突章 5500-7500，过渡/铺垫章 4500-5000。
- [ ] **与前面剧情无矛盾**：人物性格、已揭示信息、时间线、人物关系一致。
- [ ] **线索同步更新**：新抛出的线索、回收的线索、新人物/地点/设定已更新到 `docs/伏笔与线索追踪.md`。
- [ ] **给下一章留的钩子明确**。
- [ ] **数据库同步**：已执行 `cd server && node sync_chapter.js <章节号>`，提交包含 `server/writer.db`。

---

## 三、一句话判断

如果一章写完后，说不清它推动了什么、改变了什么、为后面埋了什么——这一章就是废章，要么改，要么删。
